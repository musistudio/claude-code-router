## `ccr` vs. the desktop command

CCR has two related commands:

| Command | Source | Primary use |
| --- | --- | --- |
| `ccr` | npm package `@musistudio/claude-code-router` | Electron-free; provides the browser management UI and model gateway, and launches agents from profiles. |
| `ccr-app` | CCR desktop application | Profile launcher generated by the desktop app; commands copied from Agent Config cards use this name. |

Both distributions read the same local configuration directory, but do not mix up the command names. Use the desktop app when you need the tray, desktop notifications, automatic updates, and desktop-only browser integrations; use the npm CLI for desktop-free deployments or when an external process manager supervises the service.

## Install, upgrade, and uninstall

The CLI requires Node.js 22 or newer:

```sh
node --version
npm install -g @musistudio/claude-code-router
ccr --help
```

Upgrade and uninstall:

```sh
npm install -g @musistudio/claude-code-router@latest
npm uninstall -g @musistudio/claude-code-router
```

Uninstalling the npm package does not delete CCR's local configuration or databases.

If the installation succeeds but the command is not found, run `npm prefix -g`, confirm npm's global binary directory is on `PATH`, then open a new terminal.

## First start

Start CCR in the background and open the management UI:

```sh
ccr ui
```

On SSH or headless environments:

```sh
ccr ui --no-open
```

Then complete the configuration in this order:

1. Add a provider and at least one model.
2. Create a CCR client key on the **API Keys** page for gateway access.
3. Set the default model, routing rules, and fallbacks as needed.
4. Confirm the gateway is running on the **Server** page.
5. Point the client Base URL at the gateway address shown in the UI.

The management UI defaults to `http://127.0.0.1:3458`; the model gateway defaults to `http://127.0.0.1:3456`. The management token and CCR client keys are two independent credentials: the former protects UI / RPC, the latter authenticates model requests.

## Service command summary

| Command | Mode | Purpose |
| --- | --- | --- |
| `ccr start` | Background | Starts the management service and model gateway, and prints the authenticated management URL. |
| `ccr ui` | Background | Reuses or starts the background service and opens a browser. |
| `ccr stop` | One-shot | Stops the background service started by `start` or `ui`. |
| `ccr serve` | Foreground | Runs in the current terminal; suitable for viewing logs or handing off to a process manager. |
| `ccr web` | Foreground | Alias of `serve`. |
| `ccr <profile-name-or-id>` | Foreground | Launches an enabled Agent Config profile. |

## `ccr start`

```text
ccr start [--host <host>] [--port <port>] [--open|--no-open] [--gateway|--no-gateway]
```

| Option | Description |
| --- | --- |
| `--host <host>` | Management service listen address, default `127.0.0.1`. `--host=value` is also accepted. |
| `--port <port>` | Preferred management service port, default `3458`. `--port=value` is also accepted. |
| `--open` | Open a browser after startup. |
| `--no-open` | Do not open a browser. |
| `--gateway` | Explicitly request model gateway startup; this is the default behavior. |
| `--no-gateway` | Start only the management service, without bringing up the model gateway during startup. |

If the preferred port is occupied, CCR keeps trying following ports and prints the actual URL. The port must be an integer between `1` and `65535`.

## `ccr ui`

```text
ccr ui [--host <host>] [--port <port>] [--open|--no-open] [--gateway|--no-gateway]
```

`ui` uses the same background service as `start`, but opens a browser by default. The management URL contains the `ccr_web_token` query parameter; treat the full URL as a password and do not paste it into logs, tickets, or public screenshots.

## `ccr serve`

```text
ccr serve [--host <host>] [--port <port>] [--open|--no-open] [--gateway|--no-gateway]
```

`serve` stays in the foreground and shuts down the management service and configured services on `SIGINT` or `SIGTERM`. Prefer it when troubleshooting startup errors, because errors are printed directly to the current terminal.

`ccr stop` only manages background services. A foreground `serve` should be stopped from its terminal or by an external process manager.

## Background service reuse rules

`start` and `ui` write the process ID, URL, and a private service token into `service.json`. On the next invocation, CCR first verifies the process and RPC identity:

- A valid service is reused directly; a second background process is not started.
- Newly passed host, port, and `--no-gateway` options do not reconfigure an already running process.
- If the new command requires the gateway, CCR tries to start it inside the existing management process.
- If the state file is stale or the process has exited, CCR cleans up the old state and starts a new service.

To change listener settings, stop first:

```sh
ccr stop
ccr start --host 127.0.0.1 --port 3458
```

## Launch Agent Config profiles

Create and enable a profile under **Agent Config** first, then use:

```text
ccr <profile-name-or-id> [cli|app] [-- <agent arguments>]
```

Examples:

```sh
ccr "Codex - Work"
ccr "Codex - Work" app
ccr "Claude - Review" cli -- --model sonnet
ccr profile-id -- --help
```

The rules:

- `--cli` and `--app` can replace the positional `cli` / `app` form.
- Put the agent's own arguments after `--` to avoid conflicts with CCR options or the surface name.
- When the surface is omitted, Claude Code, Codex, and Grok CLI default to CLI; ZCode defaults to App.
- Grok supports CLI only; ZCode supports App only.
- Claude App and ZCode App do not support extra agent arguments.
- Launching an App requires the corresponding desktop application installed locally and a graphical session in the current environment.
- Only enabled profiles can be launched. Use the profile ID when the name is ambiguous.

Most profiles require the CCR gateway to be running. Grok CLI is the exception: if no service exists, it can automatically start a managed temporary shared service and shut it down after the last Grok session exits.

## Configuration and data locations

| Platform | Configuration directory |
| --- | --- |
| macOS / Linux | `~/.claude-code-router` |
| Windows | `%APPDATA%\claude-code-router` |

Common files and directories:

| Path | Purpose |
| --- | --- |
| `config.sqlite` | Current application configuration. |
| `app-data/` | Runtime data such as API keys, usage, request logs, and certificates. |
| `service.json` | Background CLI service state and private token. |
| `gateway.config.json` | Generated gateway runtime configuration. |
| `profiles/` | Per-Agent-Config isolated files. |
| `bin/` | Agent launch wrappers generated by CCR. |

Do not edit or copy live SQLite files while CCR is running. Prefer **Settings → Export data**; stop the CLI and desktop app before taking a file-level backup.

## Environment variables and remote access

The public management authentication variables are:

| Variable | Description |
| --- | --- |
| `CCR_WEB_HOST` | Management service listen address used when `--host` is omitted. |
| `CCR_WEB_PORT` | Management service port used when `--port` is omitted. |
| `CCR_WEB_AUTH_TOKEN` | Fixed management UI / RPC token; without it the process generates a random token. |

Listening on `0.0.0.0` exposes the management UI to the LAN or external networks. Only configure this when genuinely needed, and combine it with a fixed strong token, a host firewall or private network, and TLS provided by a trusted reverse proxy.

The model gateway additionally requires separately created CCR client keys. Upstream provider credentials are stored in the local data directory, so both the directory and its backups should be protected as sensitive data.

## Process manager example

In production, use `ccr serve --no-open` and let an external manager handle restarts and logs. The start command should at least pin the working user, `HOME`, the listen address, and `CCR_WEB_AUTH_TOKEN`. Do not also run a background service created by `ccr start`, otherwise you may end up with two management ports or two processes competing for the same configuration.

## Troubleshooting

### The UI opens, but `/health` or model requests fail

The management service can run without a usable model gateway. Add a provider and model, create a CCR client key, then start or restart the gateway from the **Server** page. Use `ccr serve` to see startup errors.

### The management port shifted

Port 3458 was already occupied, so CCR used the next available port. Trust the URL printed by the command; to pin a port, stop the conflicting process first.

### Agent Config profile not found

Confirm the profile is enabled and check for duplicate names. CCR matches by ID, name, case-insensitive name, and sanitized name; when multiple results match, the ID must be used.

### The launcher does not exist

Open CCR once or re-save that Agent Config profile so CCR regenerates the launch wrappers under `bin/`.

### The background service cannot be stopped

Run `ccr stop` first. If the state file is already stale, the command cleans it up and reports that the service is not running. A foreground `ccr serve` is not managed by `ccr stop`; return to its terminal or process manager to stop it.
