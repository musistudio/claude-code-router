## Q&A

### 接入与观测

#### Q: Agent 没有走 CCR，从哪里查？

A: 服务状态、Agent 启动方式、Agent 配置的应用状态和作用范围，四项中任何一项不对，请求都会绕过 CCR。按顺序检查：

1. 确认 CCR 服务正在运行。
2. 确认 Agent 是从 CCR 启动的，而不是直接打开的。
3. 在 Agent 配置页确认配置已应用，且作用范围覆盖当前项目。

#### Q: 观测面板没有 Agent 执行链路怎么办？

A: 观测开关没有打开。到 **设置 → 日志与观测** 开启 **请求日志** 和 **Agent 观测**，然后重新发起一次 Agent 任务。观测面板只记录开关打开之后新执行的任务，包含步骤、工具调用、工具结果和耗时。

### 连接与认证错误

#### Q: 供应商返回 401 或 403 怎么处理？

A: 这是凭据问题，不是路由问题。依次检查：

1. API Key 是否正确、凭据是否已启用。
2. Base URL 和协议是否与供应商要求一致。
3. 是否需要额外请求头。

改完后用供应商页面的模型连通性检查验证。

#### Q: 出现 `model not found` 怎么排查？

A: 路由解析出的模型名不在供应商的模型列表里。模型名出现在三个地方：供应商模型列表、路由配置选中的模型、Agent 配置里的模型。逐一对比，把不一致的地方改过来。

#### Q: 某条 Key 一直失败怎么办？

A: 先在请求日志里按凭据筛选，确认失败是否集中在这条 Key 上。如果是，问题出在 Key 本身：检查它的额度、权限和供应商后台的账号状态，而不是改 CCR 配置。

#### Q: 请求超时怎么排查？

A: 先看请求日志里的耗时和错误信息，判断超时发生在哪一段。上游服务延迟、Fusion 工具执行耗时和 timeout 设置都可能导致超时；如果耗时集中在工具调用，调大对应的 timeout。

### 路由与成本

#### Q: 请求命中了错误模型怎么办？

A: 先在请求日志里对比 `request model`、`resolved provider` 和 `resolved model`，确认请求被路由到了哪里。再到路由配置页检查规则顺序、匹配条件和回退（fallback）规则——规则按顺序匹配，顺序或条件不对就会命中错误模型。

#### Q: 成本突然变高怎么定位？

A: 用请求日志定位，不要猜。按模型、供应商或凭据筛选日志，查看 token 组成、请求体大小和最终命中的模型，找出是哪类请求贡献了增量。

### Fusion 与工具调用

#### Q: Fusion 没有调用工具怎么办？

A: 通常是工具链路中某一环没配好。依次检查：

1. Fusion 工具是否已启用。
2. Vision model 或搜索服务 Key 是否已配置。
3. MCP 的 Discover tools 是否返回了工具。
4. timeout 是否设置得太短。

### Bot 与消息

#### Q: Bot 收不到消息怎么排查？

A: 消息链路中任何一环断开都会表现为收不到消息。依次检查 Bot 开关、消息转发设置、平台 Token、回调配置，以及 Agent 是否是从 CCR 打开的。
